# ⏱ Time Awareness — AI 时间差系统 v2.0

[![License](https://img.shields.io/badge/license-MIT-blue)](LICENSE)
[![Bash](https://img.shields.io/badge/bash-5.0%2B-green)]()

让 AI Agent 拥有「现在几点、今天几号」的时间感知能力。

**核心理念**：时间戳是记忆的原生维度。每条记忆自带时间标签，AI 通过对比时间戳自然感知信息的新旧与时效性。活跃度等行为规律由记忆系统动态管理，不由脚本硬编码。

---

## 快速开始

```bash
bash temporal_context.sh
# 输出: 🕐 北京时间 2026-06-19 20:47 CST
```

将输出注入 AI Agent 的 system prompt 即可。

---

## 为什么需要时间差？

当前大多数 AI Agent 有两个时间盲区：

| 盲区 | 表现 | 根因 |
|------|------|------|
| **时钟盲** | 不知道现在几点，靠上下文语义猜 | 没有主动查时间机制 |
| **衰减盲** | 上下文里新旧消息权重相同 | 没有时间戳降权 |

典型翻车：上午 11:00 对用户说「晚安」——上下文里有凌晨的语义，但 AI 没查时间。

---

## v2.0 设计哲学

### 从「注入」到「原生」

**v1：** 时间信息靠脚本注入一行文本
```
🕐 北京时间 20:47 CST | 用户活跃度:中高
```
用完就丢，不参与记忆。

**v2：** 时间戳跟随每条记忆条目，AI 检索时自然带时间：
- `📰 [2026-06-14] 美伊协议...` → 嗯，几天前了
- `📝 [2026-06-19] 写作偏好...` → 今天刚记的

AI 看到时间戳对比，自然知道新老、自然判断信息时效性。**不需要额外告诉 AI "现在是几点"**——`temporal_context.sh` 报一下就够了，AI 通过时间戳对比自己理解时间上下文。

### 活跃度去哪了？

活跃度不再由脚本硬编码，而是作为知识条目写入记忆系统：

1. 活跃度表写成 `📌 [时间感知]` 记忆条目
2. AI 的 bge-m3 检索可自然召回
3. Sleep Loop（夜间离线聚类）自动抽象成知识图谱节点
4. 行为变化时，Sleep Loop 会动态更新，无需手动改脚本

## 架构

```text
每轮对话
  │
  ▼
① temporal_context.sh → 🕐 北京时间 HH:MM（仅时间，无活跃度）
② bge-m3 检索记忆 → 带回带时间戳的记忆条目
  │
  ▼
AI 自然感知：
  - 这个记忆是 X 天前的，可能过期了
  - 这条是今天的，可信度高
  - 活跃规律在那个条目里，需要时能查到
  │
  ▼
Sleep Loop（凌晨 2:00）
  - 扫描 [时间感知] 标签条目
  - 聚类抽象 → KG 节点
  - 行为变化 → 更新活跃规律
```

### 联动设计

```text
时间差系统（薄层）
  │
  ├─→ 记忆系统  时间戳作为检索权重条件（不修改 RAG 索引）
  ├─→ 串线防御  时间距离降权旧消息
  └─→ 知识图谱  活跃时段偏移 → Sleep Loop 自动更新
```

### 记忆检索的时间权重

凌晨 2:00 检索 → 凌晨相关的记忆权重 ×1.2
下午 5:00 检索 → 工作相关的记忆权重 ×1.2
实现方式：不改 RAG 索引结构，在检索结果返回后加时间重排序层。

### 串线防御

消息时间差 > 2h → 旧消息权重 ×0.3（话题可能已切换）
消息时间差 > 6h → 旧消息权重 ×0.1（几乎是新对话）

## 自定义

**v2.0 不再提倡编辑脚本修改行为。**

要修改活跃度时钟或时间感知规则，直接在 AI 的记忆系统中写入新的 `📌 [时间感知 YYYY-MM-DD]` 条目。Sleep Loop 会自动处理聚类和抽象。

如需修改脚本本身（如时区、输出格式），编辑 `temporal_context.sh` 即可。
